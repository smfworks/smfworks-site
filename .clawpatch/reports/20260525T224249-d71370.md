## high: @types/stripe v8 conflicts with stripe v20 — built-in types already ship

id: fnd_sig-feat-config-7528cb5b98-c84cd_549068010e
category: api-contract
confidence: high
triage: contract-mismatch
status: open
feature: Project config package.json (feat_config_7528cb5b98)

evidence:
- package.json:21
- package.json:29

stripe@^20.4.1 ships its own TypeScript declarations starting from v9. The devDependency @types/stripe@^8.0.416 provides type definitions for the Stripe v8 API surface, which is fundamentally different from the v20 API (different method names, parameter shapes, and response types). TypeScript will resolve these competing definitions unpredictably, likely preferring the ambient @types/stripe declarations over the bundled ones, producing incorrect type inference for Stripe API calls. This can mask real bugs — e.g., passing wrong parameter types to charge creation or payment-intent calls — until they surface at runtime against the live Stripe API.

recommendation:
Remove @types/stripe from devDependencies entirely; stripe v20 already provides accurate types. If any code imports types from 'stripe' directly, TypeScript will resolve them correctly once the ambient declarations are gone.

test analysis:
The project has no test script defined, so there is no automated validation that Stripe API calls compile or behave correctly.

suggested regression test:
Add a type-level compile check that instantiates a Stripe client and calls a single API method (e.g., stripe.customers.list({limit: 1})) to ensure types resolve from the bundled declarations.

minimum fix scope:
Remove @types/stripe from devDependencies, run tsc to verify no new errors appear, and verify Stripe imports resolve correctly.

## medium: Missing forceConsistentCasingInFileNames risks cross-platform build failures

id: fnd_sig-feat-config-0c1c23856a-b5e15_738f25db45
category: build-release
confidence: high
triage: risk
status: open
feature: Project config tsconfig.json (feat_config_0c1c23856a)

evidence:
- tsconfig.json:2-23

The compilerOptions section (lines 2-23) does not include `"forceConsistentCasingInFileNames": true`. This option is not part of `strict` and must be set explicitly. Without it, imports like `import X from './MyComponent'` where the actual file is named `mycomponent.tsx` will pass type-checking on case-insensitive filesystems (macOS, Windows) but fail at build time on case-sensitive filesystems (Linux CI, production deployments). Next.js production builds and Vercel deployments run on Linux, making this a concrete risk.

recommendation:
Add `"forceConsistentCasingInFileNames": true` to the `compilerOptions` block to catch casing mismatches during local development and CI.

test analysis:
No test suite is configured (detected commands show null for test). Even if tests existed, casing bugs only manifest on case-sensitive filesystems, so a macOS-only test run would not catch them.

suggested regression test:
Run `tsc --noEmit` in CI on a Linux runner; with the option enabled, any existing casing mismatches in the codebase will be surfaced as compile errors.

minimum fix scope:
Add a single line to tsconfig.json: `"forceConsistentCasingInFileNames": true` inside compilerOptions.

repro:
On macOS, create a file `Foo.tsx` and import it as `import Foo from './foo'`. `tsc --noEmit` will succeed. The same code will fail on Linux with a module-not-found error.

## low: design:export silently swallows failures from the first npx command

id: fnd_sig-feat-config-7528cb5b98-b246c_71b9cdc221
category: build-release
confidence: medium
triage: risk
status: open
feature: Project config package.json (feat_config_7528cb5b98)

evidence:
- package.json:11

The design:export script chains two npx commands with &&. If the first export (--format tailwind) fails, the second (--format dtcg) is never run, but no explicit error handling or exit-code propagation exists beyond what && provides. More critically, npx downloads and caches @google/design.md on each run if not locally installed, making build reproducibility dependent on network availability and npm registry state. A CI pipeline invoking this script may succeed with only a partial export without any visible failure unless set -e or equivalent is active in the shell environment.

recommendation:
Either add @google/design.md as a devDependency for reproducible installs, or wrap the npx calls with explicit error handling (e.g., set -e; npx ... && npx ...). Consider running both exports unconditionally or capturing exit codes.

test analysis:
No post-export validation step checks that both output files were produced and contain valid content.

suggested regression test:
Add a script that asserts both design-export.tailwind.json and design-export.dtcg.json exist and are non-empty after design:export runs.

minimum fix scope:
Add error handling to the design:export script or make @google/design.md a declared dependency.
